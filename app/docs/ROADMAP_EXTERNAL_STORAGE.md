# ROADMAP_EXTERNAL_STORAGE.md

## Overview

This document defines the roadmap and architecture for integrating external storage and publishing targets into the platform.

The goal is to enable users to **send generated image assets into third-party systems** such as:

- Miro
- Canva
- Figma
- Jira
- ServiceNow
- S3 / object storage
- Custom backends

The system must support:

- Target discovery (browse, search, select)
- Authentication and connection management
- Asset publishing (sync, attach, place)
- Repeatable workflows and automation

---

## Core Architecture

### Key Concepts

#### 1. Asset
An image generated by the system, including metadata, variants, and binary source.

#### 2. Connection
An authenticated link to a third-party system using OAuth, API keys, presigned URLs, or custom auth.

#### 3. Target Context
A specific destination location such as a Jira issue, Miro board, ServiceNow record, Figma file, or S3 bucket/prefix.

#### 4. Publish Job
An asynchronous operation that sends an asset to a selected target.

---

## Publish Flow

```text
Asset → Destination → Connection → Target Context → Optional Placement → Publish Job
```

---

## Destination Types

### 1. Attachment Targets
These store images as files attached to records.

Examples:

- Jira issue attachments
- ServiceNow record attachments

### 2. Canvas / Placement Targets
These insert images into visual environments.

Examples:

- Miro boards and frames
- Figma files, pages, and frames
- Canva designs

### 3. Asset Library Targets
These store reusable assets.

Examples:

- Canva asset library
- CMS or DAM systems

### 4. Object Storage Targets
These store images as objects.

Examples:

- Amazon S3
- S3-compatible storage such as Cloudflare R2, MinIO, Wasabi, or Backblaze B2

## Context Discovery

Each connector must implement:

- Browse
- Search
- Resolve from URL or ID

### UX Requirements

Users must be able to:

- Browse hierarchy, for example workspace → board
- Search by name or reference, for example `ABC-123`
- Paste URLs, for example a Jira issue link or Miro board link
- Select recent and favourite targets

## S3 Integration (First-Class Target)

### Why S3 Matters

S3 should be treated as a first-class target because it is:

- A system of record
- A common enterprise requirement
- A flexible storage backend
- Often customer-owned

---

---

## S3 Upload Modes

### 1. Managed Upload
The backend uses AWS credentials or an IAM role to write directly to the bucket.

Best suited to:

- Enterprise environments
- Controlled infrastructure
- Scheduled or backend-driven publishing

### 2. Presigned Upload
The system obtains a presigned URL and uploads without exposing direct credentials.

Best suited to:

- Browser or local-first apps
- Customer-owned buckets
- Reduced credential sharing

### 3. S3-Compatible Mode
The system supports object stores with S3-compatible APIs.

Examples:

- Cloudflare R2
- MinIO
- Wasabi
- Backblaze B2

---

## S3 Features

### Required

- List buckets
- Browse prefixes
- Upload objects
- Validate permissions

### Recommended

- Multipart upload for large files
- Checksum validation
- Retry handling

### Optional Enhancements

- Object tagging
- Metadata injection
- Encryption settings
- Storage class selection
- Cache-Control headers
- Content-Disposition headers

## Naming Strategy

Avoid making users manually type object keys.

Support templates such as:

```text
{project}/{yyyy}/{mm}/{assetName}-{variant}.{ext}
{customer}/{workflow}/{date}/{checksum8}.{ext}
generated/{appId}/{yyyy-mm-dd}/{assetId}/{filename}
```

### UX Requirement

Always show a live preview of the resolved object key, for example:

```text
customer-a/social/2026/04/banner-instagram-portrait.png
```

## Automation and Rules (Future)

```ts
type TargetResolverRule = {
  id: string;
  destinationType: string;
  condition: {
    filenamePattern?: string;
    tags?: string[];
    projectId?: string;
  };
  resolution: {
    strategy: "fixed" | "search";
    targetContextId?: string;
    queryTemplate?: string;
  };
};
```

These rules enable workflows such as:

- Route all assets with a specific filename pattern to a fixed S3 prefix
- Send assets tagged with a campaign name to a matching Miro board
- Attach image variants to a Jira issue found from a metadata field

## MCP (Model Context Protocol)

### Role of MCP

MCP should **not** be the core integration architecture.

It should be an **AI-facing interface layer** over the connector platform.

That means the core system still owns:

- Connections
- Target discovery
- Publishing logic
- Retry handling
- Audit history

MCP then exposes those capabilities to AI clients.

---

## MCP Capabilities

### Resources

- `asset://{id}`
- `connection://{id}`

### Tools

- `search_assets(query)`
- `search_targets(destinationType, query)`
- `browse_targets(destinationType, parentId)`
- `publish_asset(assetId, targetContextId)`
- `get_publish_status(jobId)`

- MCP server

---

## Key Design Principles

1. Do not treat every target as a simple file upload
2. Separate connection, target context, and publish action
3. Support browse, search, and direct URL input
4. Make S3 first-class rather than hiding it behind a generic backend abstraction
5. Build MCP as an overlay, not the foundation

---

## Recommended Architecture Summary

The system should evolve into a **unified publishing platform** where:

- Assets are generated once
- Targets are discoverable and reusable
- Publishing is consistent across systems
- Storage targets such as S3 are supported alongside collaboration tools
- AI workflows are enabled through MCP without coupling core logic to AI clients

---

4. Add saved targets, recent targets, and favourites to the UX
